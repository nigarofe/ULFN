### Introduction
The "Keywords Exercise" document classification is dedicated to documents which the main goal is to practice iteratively how to answer open ended questions or elaborate on topics.

The main problem I'm trying to solve is how to evaluate consistently open-ended answers, how to say if one answer is better or worst than another.

You can theoretically use an LLM to evaluate that, however, which system instructions should you give? And how do you know how the specific system instructions you gave will interact with the specific LLM you are using? This makes both the prompt and LLM source of very hard to predict bias, require internect connection and all the software implementation complexity that comes with interacting with APIs.

By scoring the answer by keywords matching, unpredictable bias is removed and the software implementation is simpler.

Besides, instead of the student being required to formulate an answer from start each iteration, he can remember the keywords and them build the structure around it, which will be specially useful when integrated with spaced repetition.

**Instant feedback**
By displaying the score of the student's answer in real time, the student is implicitly encouraged to remember the keywords.

**Weighted Scoring**

**Disadvantages**
This way of scoring can be gamed by the student if he just says all the keywords without any structure, or just keep trying to see the instant feedback. However, as this software is designed for a single user, it's expected that he will be honest with himself.

### Introduction

The **"Keywords Exercise"** is a framework designed for iterative practice in answering open-ended questions. It focuses on helping learners refine their ability to articulate complex topics through structured recall.

### The Problem

Evaluating open-ended answers consistently is notoriously difficult. Determining why one subjective answer is "better" than another often leads to inconsistent results.

While Large Language Models (LLMs) can theoretically handle evaluation, they introduce significant hurdles:

* **Unpredictable Bias:** System instructions interact differently across models, making "fair" scoring a moving target.
* **Technical Complexity:** Integration requires stable internet connections, API management, and complex software architecture.
* **Latency:** The delay in API calls disrupts the flow of "instant" practice.

### The Solution: Keyword-Based Scoring

By transitioning to a keyword-matching algorithm, we remove unpredictable AI bias and significantly simplify the implementation. This approach shifts the student's cognitive load: rather than struggling to structure a response from scratch every time, the student focuses on mastering the core concepts (keywords). This makes the method exceptionally effective when paired with **Spaced Repetition Systems (SRS)**.

---

### Pedagogical Value

#### **Instant Feedback Loop**

Real-time score updates provide immediate reinforcement. This creates a "gamified" environment where the student is incentivized to bridge the gap between their current draft and the required key concepts.

#### **Weighted Evaluation**

Not all concepts are created equal. The system allows for **Weighted Scoring**, where specific "anchor keywords" are required for a passing grade, while others act as supplementary points. This ensures a more nuanced and accurate assessment of the student’s understanding.

#### **Addressing Limitations**

A keyword-based system can, in theory, be "gamed"—a student could simply list words without logical structure. However, because this tool is designed for **self-directed learning**, the user is the primary beneficiary. The system assumes "integrity by design"; the student uses the keywords as a scaffold to build better mental models, not just to "beat" the code.

### Overview
The Regex Grading System is an automated, text-based evaluation tool designed to assess student responses iteratively. By scanning a student's text for specific mandatory vocabulary, concepts, and logical connectives, the system provides immediate, objective feedback on their writing. 

### 🧠 The Learning Logic (Pedagogical Value)

The core philosophy behind this feature is that **writing is an iterative process of structuring thought**. This system goes beyond simply checking right or wrong answers; it trains students to articulate complex ideas professionally.

* **Promotes Domain-Specific Vocabulary:** By setting rules that require specific technical terms (e.g., "microestrutura", "porosidade gasosa"), the system forces students to abandon colloquialisms and adopt the precise language of their discipline.
* **Encourages Logical Structuring:** The system isn't just for technical words; it can be programmed to require logical connectives (e.g., "portanto", "entretanto", "além disso"). This trains students to build cohesive arguments with clear introductions, contrasts, and conclusions.
* **Facilitates Iterative Refinement:** The inclusion of an "Iterations"  and "Attempts"  section reflects a growth-mindset approach. Students are encouraged to rewrite and refine their answers, learning from the visual feedback of their previous attempts.
* **Immediate, Unbiased Feedback:** Students instantly see which concepts they successfully integrated and which are missing, reducing the feedback loop from days to seconds.

### ⚙️ How It Works (Technical Architecture)

The system works by parsing a Markdown document into specific functional blocks and evaluating the student's text against predefined rules.

#### Document Parsing
The post-processor scans the HTML/Markdown and extracts four critical sections using regular expressions:
* **Student Prompt:** The question or task given to the student.
* **Iterations:** The historical log of the student's submitted answers.
* **Regex (Rules):** The section containing the grading criteria.
* **Attempts:** A log of the submission attempts.

#### Rule Processing
When the system reads the `## Regex` section, it looks for list items containing the grading rules. 
* It extracts the required match count and the comma-separated list of target words.
* The words are automatically converted to lowercase, trimmed of excess whitespace, and cleverly **sorted by length in descending order**. This sorting ensures that longer, more specific phrases (e.g., "porosidade de contração") are matched before shorter, overlapping single words.

#### Evaluation and Rendering
For every student iteration found, the system performs the following:
* **Text Matching:** It builds a dynamic regular expression for each target word to ensure it only matches whole words (avoiding partial matches inside other words) and ignores HTML tags.
* **Scoring:** It counts how many target words from a rule were found in the text, capping the score at the rule's required count so students can't game the system by repeating the same word.
* **Visual Highlighting:** Every successfully matched word is dynamically wrapped in an HTML `<span>` to highlight it in bold green text with a light green background (`#e6ffe6`).
* **Score Display:** A final score (`Score: X / Y`) is generated for each iteration. If the student meets or exceeds the total required matches, the score turns green.

### 📝 Instruction Manual: Writing Grading Rules

To utilize the Regex Grading engine in your exercises, you must configure the rules using a specific syntax. 

**Section Placement**
All rules must be placed under the specific heading `## Regex`.

**Rule Syntax**
Each rule must be a list item (starting with a `-`) followed by the required quantity of words, and then the comma-separated words.
* *Format:* `- [Required Quantity] [word 1], [word 2], [phrase 3]`
* *Example:* `- 2 tenacidade à fratura, tenacidade ao impacto, ductilidade`. (This means the student must use at least 2 of these 3 terms to get the points).

**Adding Comments**
Any text inside the `## Regex` section that does not strictly follow the `- [number] [words]` format is ignored by the parser. You can freely use headings, bold text, or regular paragraphs to categorize your rules (e.g., separating "Defeitos e Descontinuidades" from "Conectivos").
